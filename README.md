# BrokerMindAI Launchpad

Marketing landing page and waitlist for **BrokerMindAI** — AI decision
intelligence for mortgage brokers, private lenders, B lenders, credit unions,
and modern lending teams.

Built on [TanStack Start](https://tanstack.com/start) (React 19 + Vite 8) with
Tailwind CSS v4 and Framer Motion.

## Quick start

```bash
bun install
bun run dev        # http://localhost:8080
bun run build      # production build
bun run preview    # preview the production build
bun run lint       # eslint
bun run format     # prettier
```

> Requires [Bun](https://bun.sh) 1.x. The lockfile is `bun.lock`.

## Environment variables

Values live in `.env` (git-ignored) and are inlined at build time by Vite
when prefixed with `VITE_`.

| Variable | Scope | Purpose |
| --- | --- | --- |
| `VITE_SUPABASE_URL` | Client + server | Supabase project URL |
| `VITE_SUPABASE_ANON_KEY` | Client + server | Supabase anon/publishable key |
| `SUPABASE_URL` | Server only | Supabase project URL for server routes |
| `SUPABASE_ANON_KEY` | Server only | Anon/publishable key for server-side auth checks |
| `SUPABASE_SERVICE_ROLE_KEY` | Server only | Service role key — bypasses RLS, never expose to the client |
| `MAILERLITE_API_KEY` | Server only | MailerLite waitlist integration |

## Project structure

```
public/            Static assets served as-is (robots.txt, favicon, etc.)
src/
  assets/          Bundled images imported from components
  components/ui/   Presentational UI primitives (accordion, sonner toaster)
  lib/             App-wide utilities (cn, error page, error capture)
  services/        Backend-facing service layer
    waitlist.ts    Waitlist abstraction — swap the transport, not the UI
  routes/          File-based routes
    __root.tsx     Root layout, head metadata, error/404 boundaries
    index.tsx      Landing page
    sitemap[.]xml.ts  Dynamic sitemap at /sitemap.xml
  router.tsx       TanStack Router instance
  server.ts        SSR entry (wraps TanStack Start with a safe 500 page)
  start.ts         TanStack Start middleware registration
  styles.css       Tailwind v4 entry + design tokens
```

Route files are the source of truth; `src/routeTree.gen.ts` is regenerated by
the TanStack Router Vite plugin — never edit it by hand.

## Waitlist

The waitlist form (`WaitlistForm` in `src/routes/index.tsx`) submits through
a single service function:

```ts
import { joinWaitlist } from "@/services/waitlist";

await joinWaitlist({ email, utm_source, utm_campaign, referrer });
```

`src/services/waitlist.ts` POSTs to `/api/waitlist` with the following
contract:

```
POST /api/waitlist
  body:   { email, utm_source?, utm_campaign?, referrer? }
  200:    { ok: true, alreadyJoined?: boolean }
  409:    { ok: true, alreadyJoined: true }   // duplicate email
  4xx:    { ok: false, error: string }
```

No backend is wired up yet. Until one is, the service accepts submissions
optimistically (a dev-mode warning is logged) so the marketing page works
end-to-end. Client-side email validation and UTM/referrer attribution are
always performed.

### Connecting a real backend

Pick one of the following and only touch `src/services/waitlist.ts` — no UI
changes are required.

1. **Direct provider SDK** (MailerLite, HubSpot, ConvertKit, Resend, etc.) —
   import their SDK inside `joinWaitlist` and call it after `validateEmail`.
   Do NOT ship provider secrets to the browser; use a server route for any
   provider that requires authenticated writes.
2. **Internal API** — implement `POST /api/waitlist` as a TanStack Start
   server route at `src/routes/api/waitlist.ts`. The existing service will
   call it automatically. Verify + rate-limit the endpoint on the server.
3. **Third-party form endpoint** (Formspree, Basin, etc.) — change
   `WAITLIST_ENDPOINT` at the top of `src/services/waitlist.ts` to the
   provider URL.

Whichever transport you pick, keep the response shape in the contract above
so `joinWaitlist` can distinguish new sign-ups, duplicates, and errors.

## SEO

- `public/robots.txt` — allows all crawlers.
- `src/routes/sitemap[.]xml.ts` — dynamic sitemap served at `/sitemap.xml`.
  Set `BASE_URL` once the production domain is assigned.
- `src/routes/__root.tsx` — sitewide metadata (title, description,
  Open Graph, Twitter Card, `Organization` JSON-LD).
- `src/routes/index.tsx` — page-level metadata plus `SoftwareApplication`
  and `FAQPage` JSON-LD.

When a canonical domain is available, replace the empty `BASE_URL` in the
sitemap route and swap the relative `og:url` / `canonical` values for
absolute URLs.

## Deployment

GitHub → Vercel → Production. The build targets Vercel via the `nitro`
`vercel` preset configured in `vite.config.ts`.

1. `bun run build`
2. Push to `main` — Vercel deploys automatically.
3. Set the production domain, then update `BASE_URL` in the sitemap route
   and any absolute URLs in metadata.

## Accessibility

- Semantic landmarks (`<main>`, `<nav>`, `<footer>`) and skip-to-content link.
- Focus-visible rings on all interactive elements.
- `prefers-reduced-motion` respected for Framer Motion animations and
  confetti.
- All images ship with meaningful `alt` text; decorative images use
  `aria-hidden`.

## Adding future integrations

- **Analytics** — add the script tag in `src/routes/__root.tsx`'s `scripts`
  array; wrap page-view tracking in a route effect.
- **Backend API** — add server routes under `src/routes/api/*` (TanStack
  Start server routes). Public callers should live under
  `src/routes/api/public/*` and verify the caller in the handler.
- **Auth / database** — add the client behind a service module in
  `src/services/*` so the UI stays decoupled from the transport.

Keep provider secrets out of the client bundle. Anything referenced in
client code must be prefixed with `VITE_` and safe to expose publicly.
